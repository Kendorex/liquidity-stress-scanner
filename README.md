# Liquidity Stress Sentinel — README для продолжения проекта

## 1. Что мы вообще делаем по ТЗ

Мы делаем систему **раннего выявления стресса ликвидности** для казначейства банка.

Идея проекта: собрать данные из открытых источников, превратить их в числовые сигналы и на их основе построить единый индекс стресса ликвидности.

Итоговая система должна показывать:

```text
норма / напряжение / стресс
```

и объяснять, какие факторы создают давление на банковскую ликвидность.

---

## 2. Общая логика проекта

По ТЗ система состоит из нескольких модулей:

```text
M1 — Усреднение обязательных резервов
M2 — Аукционы РЕПО ЦБ
M3 — Размещение ОФЗ
M4 — Налоговые периоды
M5 — Средства Федерального казначейства / ЕКС
```

Каждый модуль должен:

1. скачивать данные из своего источника;
2. очищать и приводить данные к единому виду;
3. считать собственные стресс-сигналы;
4. нормализовать показатели;
5. сохранять итоговую таблицу сигналов;
6. при необходимости строить график для дашборда.

В конце сигналы всех модулей должны объединяться в общий **Liquidity Stress Index / LSI**.

---

## 3. Что уже сделано

На данный момент полностью подготовлен модуль:

```text
M1 — Усреднение обязательных резервов
```

Этот модуль показывает, насколько банки держат денег на корсчетах в ЦБ больше обязательной нормы.

Экономический смысл:

```text
если фактические остатки на корсчетах сильно выше обязательных резервов,
значит банки могут заранее накапливать ликвидность;
если одновременно растёт RUONIA, это усиливает сигнал стресса на межбанковском рынке.
```

---

## 4. Что делает M1

M1 использует два источника:

```text
1. Обязательные резервы кредитных организаций
2. RUONIA
```

Модуль считает:

```text
spread = actual_corr_accounts - required_reserves_avg
```

где:

```text
actual_corr_accounts — фактические среднедневные остатки на корсчетах
required_reserves_avg — обязательные резервы, подлежащие усреднению
```

Также модуль накладывает RUONIA на каждый период усреднения и считает:

```text
mean_ruonia
max_ruonia
last_ruonia
MAD_score_spread
MAD_score_RUONIA
Flag_EndOfPeriod
M1_signal
```

Главный стрессовый паттерн M1:

```text
высокий spread + высокая RUONIA + конец периода усреднения = стресс ликвидности
```

Скрипты скачивания данных для M1.

```text
download_cbr_required_reserves.py
```

Скачивает Excel-файл с обязательными резервами.

```text
download_cbr_ruonia.py
```

Скачивает Excel-файл с RUONIA.

---

### `preparation_data/m1/`

Скрипт расчета итоговых сигналов M1.

```text
script_for_m1_signals.py
```

Он делает всё после скачивания:

1. читает сырые Excel-файлы;
2. очищает данные внутри кода;
3. рассчитывает spread;
4. накладывает RUONIA на периоды усреднения;
5. считает MAD-score за 3 года;
6. определяет стрессовый паттерн;
7. считает итоговый `m1_signal`;
8. сохраняет итоговые Excel-файлы и график.

---

### `stuff/`

Папка для файлов, которые сейчас не нужны для M1.

Туда можно временно складывать черновики, старые скрипты, тестовые файлы и материалы для будущих модулей.

## 10. Важные замечания по M1

### 1. Последний период может быть неполным

В файле ЦБ последняя строка может не иметь фактических среднедневных остатков, потому что период ещё не завершён.

В таком случае:

```text
signal_ready = 0
m1_signal = 0
```

Это нормально.

---

### 2. `flag_end_of_period`

Сейчас таблица M1 является периодической:

```text
одна строка = один период усреднения
```

Поэтому `flag_end_of_period = 1` для каждой строки.

Это не ошибка. Это означает, что строка уже отражает итог периода усреднения.

Если потом будет делаться daily-дашборд, нужно будет отдельно создать ежедневную таблицу и отметить последние 3–5 дней каждого периода.

---

### 3. MAD-score

MAD-score считается за скользящее окно 3 года.

Смысл:

```text
показывает, насколько текущий показатель отклоняется от своей исторической нормы
```

Используется для:

```text
spread
mean_ruonia
```


## 13. Важное правило по датам

По умолчанию парсеры должны скачивать **максимально доступную историю** с сайта-источника.

Не нужно вручную ограничивать период на этапе скачивания.

Правильная логика:

```text
скачиваем всю доступную историю
а нужный период для модели фильтруем уже на этапе обработки
```

---

## 14. Главный принцип проекта

Не смешивать модули между собой на этапе парсинга.

Каждый модуль сначала должен сам сформировать свой итоговый сигнал:

```text
M1_signal
M2_signal
M3_signal
M4_signal
M5_signal
```

И только потом отдельным финальным скриптом эти сигналы будут объединяться в общий индекс ликвидности:

```text
LSI = Liquidity Stress Index
```
